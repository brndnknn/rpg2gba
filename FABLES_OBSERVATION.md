# Fable's Observation — Project Critique (2026-06-09)

A point-in-time critique of rpg2gba: current state, the plan, the good, the bad, the ugly, and suggestions. Written by the build agent (Fable 5) at the user's request. Snapshot basis: Phases 0–3 complete; Phase 4 §9 #2 gate passed (Opus 4.8, frozen prompt); deterministic pre-filter built (~28% of events); bulk run live and limit-stalled at CEs + 7/199 maps, 87 spawns, $16.93 equivalent cost, 241 unhandled entries.

## Verdict up front

This is an unusually well-run hobby project — the engineering discipline is real, not aspirational, and the feasibility risk was retired early and cheaply. The thing that will actually hurt isn't quality, it's **throughput arithmetic**: at the observed pace, the Phase 4 bulk run takes ~3 months of calendar time, and the plan documents haven't been updated to confront that. Plus there's accumulating debt in three places: the unhandled queue, the uncommitted working tree, and a couple of internal contradictions between decisions and plans.

## The Good

- **The discipline actually binds.** Fail-loud, one-source-of-truth, idempotence, and the review gates aren't decorative — the record shows them working: the registry rejecting a script-switch flag exposed the EV011 drop; the live memo confirmation caught the `Map{NNN}_` label-rewrite bug; the false-positive rate-limit fix shipped with regression tests. Every claimed completion has test counts attached. This is rare and it's the project's biggest asset.
- **The rung-1/2/3 ladder was excellent risk management.** Before investing in calibration, the project proved poryscript-transpile → fork-assembly → in-game-on-Delta with one NPC. The core "can a converted event run on hardware" risk is dead for the price of a few spawns. Most projects like this find out at integration time.
- **The deterministic pre-filter is the best post-gate decision.** Seven classifiers converting ~1,000 events (~28%) from Opus spawns into free lookups, while keeping the output indistinguishable downstream (same compile-gate, same mint, same queue channel). The empiricism around it was also right: validating the `\sign` shape against frozen-Opus output before building, measuring that plan targets were inflated (41% claimed → 26–28% real), and dropping Classifier 5 when the corpus said no clean item balls exist.
- **The dedup levers were measured, not assumed.** Prompt cache confirmed live (`cache_read=13666` per spawn), memo hit confirmed cross-map, and the confirmation run caught a real assembly-class bug. The Sonnet-vs-Opus comparison was a properly controlled snapshot/restore experiment with a decisive, concrete differentiator (EV007's `copyvar` vs `addvar` fidelity loss).
- **The two-agent boundary held.** Five-plus weeks in, no evidence of the build agent hand-converting events or hot-patching prompts mid-run. The freeze discipline (gate note that prompt changes require recalibration) is being honored.

## The Bad

- **The roadmap is now fiction in its most important section.** ROADMAP Phase 4 still describes the local-first Ollama Stage B / Claude Stage C strategy and a "$0–20, maybe $20–80" cost picture. Reality: Ollama was (rightly) rejected for silent-wrong risk, the strategy is single-model Opus-for-everything, and measured cost is **$0.195/spawn**. The roadmap calls itself a living document; it died around June 3rd. Anyone (including future sessions) reading it for "what next" gets the wrong plan.
- **Opus spawns are being spent on STRIP-dispositioned content.** The first `unhandled.jsonl` entry is `CommonEvent 4 — GTS/WT`, an online feature Phase 0 explicitly decided to strip. The big online CEs (GTS/VT/LOBBY, 185–219 commands each) are exactly the most expensive spawns with the least value — they got converted-and-mostly-queued instead of skipped. The decorative-event skip set the precedent: a deterministic orchestrator-side STRIP list implements an existing Phase 0 decision without touching the frozen prompt.
- **The Phase 5 plan has a scope hole its own decisions created.** The 2026-06-03 decision deferred scripted move routes — **1,191 events, 3,804 commands** — "TO PHASE 5," but PHASE5_PLAN.md's sections (5.1–5.4) contain no move-route work at all. The deferred debt needs a local-id convention plus an RMXP→pokeemerald movement-vocabulary translator that nothing currently plans to build. That's not a footnote; it's plausibly weeks of work hiding between phases.
- **Q2 contradicts a recorded discovery.** The resolved design table says map names derive from `map_infos.json`, but Uranium-Specific Discoveries records that MapInfos names are stale dev labels (map 7 is named "Comet Cave," is actually Passage Cave) and that Phase 5 "must derive location identity from content + wiki, **not** from MapInfos." If readable names become the canonical constants, `MAP_COMET_CAVE` pointing at Passage Cave confuses everything downstream forever. The alias-header mechanism makes renames cheap, which softens this — but the contradiction should be resolved deliberately, not discovered at Porymap time.
- **MEMORY.md is violating its own protocol.** The "Current Phase" section is a ~4,000-word wall of single-paragraph history. It's an excellent lab notebook and a terrible session-bootstrap index — the live state ("gate passed, frozen config, bulk run at map 7/199, harness uncommitted") is buried in paragraph 30-of-40. The eviction discipline that's applied to session summaries hasn't been applied to the phase narrative.

## The Ugly

- **The throughput wall.** Do the arithmetic the plan hasn't: ~3,581 command-bearing events − ~1,000 deterministic claims − memo hits ≈ **2,300–2,500 Opus spawns remaining**. Observed: 87 spawns across 4 days of limit-capped windows ≈ 22/day. That's **~100+ days of babysitting Pro windows**, during which the same Pro budget also funds interactive build-agent sessions — they compete. At $0.195/spawn, the whole remaining run is ~$450–490 of equivalent usage; the Batch API would do it for roughly half that in days, not months. The roadmap's Stage D ("only if needed... $20–80") was written when this was hypothetical; there is now real per-spawn data and a real calendar-time cost. This is a decision that needs making, not drifting into.
- **Unhandled-queue triage debt is compounding with no tooling to match.** 241 entries after 7 maps + CEs. Maps 7 and 8 alone contributed 132. Even if early cave maps are unrepresentative, a corpus-wide projection lands somewhere between 2,000–6,000 entries, and the Phase 4 exit criterion says *every item gets a decision*. Read literally, that's another multi-week task. Most entries are by-design and cluster (warp placeholders, move-route breadcrumbs, `pbPokeCenterPC`, the ~92%-dispositioned script-call signatures) — but `triage()` as it stands doesn't auto-classify clusters, so today the criterion is unmeetable except by hand.
- **A multi-day production run is executing from an uncommitted working tree.** The bulk harness (`run_bulk.py`, `run_report.py`, the rate-limit classification, the orchestrator re-raise paths), the 06-08 false-positive fix, and the entire Phase 5 scaffold are all uncommitted. The run's behavior can't be pinned to any commit, and the same tree was edited mid-run to fix the rate-limit bug. The freeze rule (prompt/model) was honored, but harness code deserves the same discipline: commit before/between rounds.

## Suggestions, in priority order

1. **Make the throughput decision explicitly.** Three honest options: (a) accept ~2–3 months, run `--timed` continuously, overlap with Phase 5 (the current implicit plan); (b) build `AnthropicAPIBackend` — the roadmap already names it, the backend abstraction makes it small — and run the bulk via the Batch API for roughly $230–250 in days; (c) hybrid: route the ~27% trivial tier to Sonnet (the `--model` knob and difficulty classifier already exist) and keep Opus for judgment events. Even (c) alone cuts ~3 weeks. Whatever is picked, write it into ROADMAP and retire the dead Ollama strategy section.
2. **Add a deterministic STRIP skip before more budget burns.** CE 4 (GTS/WT) and the other online-feature CEs/events are already dispositioned STRIP from Phase 0; skipping them in the orchestrator is implementing a made decision, same class as the decorative-event skip. Cheap, saves the most expensive spawns, reduces queue noise.
3. **Build cluster-aware triage now, not at phase end.** Extend `run_report`/`triage()` to group `unhandled.jsonl` by (command code, script-call signature, reason) and auto-tag the by-design categories (Phase-5-deferred warps/move-routes, dispositioned UNHANDLED sigs). Then the exit criterion becomes "every *cluster* has a decision" — meetable — and genuinely novel queue items surface while the run is still going, when reacting is cheap.
4. **Automate the rung-2 assembly smoke as a recurring check during the run.** The per-file compile-gate provably cannot see assembly-class bugs (duplicate labels, undefined constants — both already happened: the SS-flag gap, the memo label bug). The rung-2 mechanism exists; a script that drops all current `.pory` output + `dump_header` into a throwaway fork branch and runs `make modern` every N maps turns "discover systemic assembly bug at Phase 7" into "discover it at map 20."
5. **Resolve the Q2/MapInfos contradiction before minting names.** Cheapest path: keep the alias header as the load-bearing mechanism, mint readable names from `map_infos.json`, but run a wiki cross-check pass (the `pokemon-uranium-wiki` skill exists for exactly this) on the ~50 maps that matter for the overworld and rename the known-stale ones before they fossilize.
6. **Add a move-route section to PHASE5_PLAN.md** — even as a stub with the open design questions (local-id convention, movement-command vocabulary, which of the 1,191 events actually need movement vs. can degrade to static NPCs). Right now that work item exists only inside a Decisions bullet, which is how scope gets lost.
7. **Commit the pile, and compress MEMORY.md.** One commit for the bulk harness + limit fixes, one for the Phase 5 scaffold. Then move the Phase 4 narrative history into `reference/memory-archive.md` and leave Current Phase as ~10 lines of live state. Both are hygiene, but both reduce the cost of every future session.
8. **One Phase 5 caution for later:** Q3 (inherit collision from the substituted metatile) + Q4 (one universal tileset) together mean walkability will be wrong on exactly the maps where it matters — caves, gyms, puzzle rooms (Gym 8's tile puzzle, the Strength boulders). Fine for the first pass, but consider adding a cheap automated reachability check (BFS from each map's entry warps to its exit warps over the emitted collision data) to the Phase 5 acceptance list. It's a soft-lock detector that costs an afternoon and replaces hours of emulator walking.

## Meta-observation

The project's quality bar is set exactly right — the get-it-right-once instinct (Opus everywhere, fail-loud, gates) is correct for a corpus you never want to re-run. The risk is no longer "will it work"; rung 3 answered that. The risk is calendar time and decision drift — the bulk run quietly becoming a 3-month background hum while plans go stale around it. Pick the throughput option deliberately and the rest is execution.
