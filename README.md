# rpg2gba

A pipeline for converting RPG Maker XP / Pokémon Essentials fan games into playable GBA ROMs, using Pokémon Uranium as the first end-to-end test case.

The pipeline tooling is the long-term deliverable. The Uranium ROM is the proof of concept.

**Personal use only.** The resulting ROM incorporates copyrighted assets and is not for distribution.

## Building a ROM (pathfinder slice: Map049 ↔ Map048 ↔ Map32)

Everything below lives in `output/uranium-build/` (gitignored) plus a handful of
gitignored generated files under `engine/` — safe to delete at any time and
regenerate with this recipe. Run the steps in order from the repo root; each one
reads files the previous step wrote.

### Prerequisites (one-time environment setup, not regenerated by this recipe)

- Python venv with the project installed editable: `.venv/bin/pip install -e ".[dev]"`.
- `ruby` on PATH (used directly, not through the venv).
- devkitARM/devkitPro installed (`engine/` is a pokeemerald-expansion checkout; needs
  `make`/`arm-none-eabi-gcc` etc. on PATH).
- A `poryscript` binary (see `RPG2GBA_PORYSCRIPT` below).
- The unpacked Uranium source tree. If you only have the distributed
  `Pokemon_Uranium_1.3.2.zip`, unzip it to get `Uranium.rgssad`, then extract
  that once with `.venv/bin/python scripts/extract_rgssad.py Uranium.rgssad <dest-dir>`.
- A `.env-paths` file at the repo root (tracked, paths only) pointing at all of
  the above:
  ```
  RPG2GBA_URANIUM_SRC=/path/to/unpacked/uranium
  RPG2GBA_POKEEMERALD=/absolute/path/to/rpg2gba/engine
  RPG2GBA_PORYSCRIPT=/path/to/poryscript
  ```
  Every command below loads this file automatically — no manual `export` needed.

### 1. Deserialize Uranium's data into JSON

```
.venv/bin/python -m rpg2gba.pipeline phase2 --clean
.venv/bin/python -m rpg2gba.pipeline phase3 --clean
ruby src/rpg2gba/rxdata_deserializer/deserialize.rb tilesets "$RPG2GBA_URANIUM_SRC/Data" output/uranium-build
```

- `phase2` runs the PBS converters (species/moves/items/abilities/trainers/…) into
  `output/uranium-build/intermediate/*.json` (feeds trainer/item lookups the
  transpiler uses in step 2; not yet compiled into the ROM — see CLAUDE.md §Phase 2).
- `phase3` runs the rxdata map deserializer → `output/uranium-build/maps/Map*.json`,
  `common_events.json`, `map_infos.json`, `system.json`.
- The `tileset` mode has no Python/CLI wrapper yet — run the Ruby deserializer
  directly. Produces `output/uranium-build/tilesets.json` (passages/priorities/
  terrain-tag oracle the tileset converter reads).

### 2. Transpile the slice's events to Poryscript

```
.venv/bin/python -m rpg2gba.conversion_agent.transpile_driver run --maps slice
```

Deterministically converts Map049/048/032 + CommonEvents (idiom classifiers +
the command→Poryscript transpiler, hand overrides from
`src/rpg2gba/conversion_agent/hand_conversions/` spliced in automatically, every
emitted symbol gated against the real fork). Writes
`output/uranium-build/scripts/Map{049,048,032}.pory` + matching `.traits.json`
sidecars, `CommonEvents.pory`, `flag_state.json`, and `transpile_unhandled.jsonl`
(the queue of anything it couldn't handle — expected to be non-empty; see
`reference/slice1_queue_readthrough.md` for what's already accounted for).

### 3. Stage + wire the maps

```
.venv/bin/python scripts/stage_slice_scripts.py --write
```

Runs the NPC sprite pass (writes gitignored `engine/**/uranium_*.gen.h` sprite
data + the `OBJ_EVENT_GFX_URANIUM_*` constants), mints map constants, wires each
map's object/warp events (NPC graphics, rock-respawn flags from the traits
sidecars, local-object-id tables), then normalizes/prunes/remaps the transpiled
`.pory` against the real wiring and runs the existence checks (no dangling or
duplicate script labels). Writes `output/uranium-build/porymap/` and
`output/uranium-build/staging/`. A non-zero exit means a check failed — read the
`FAIL:` lines before continuing.

### 4. Assemble into the vendored engine

```
.venv/bin/python scripts/assemble_pathfinder.py
```

Three passes: quantizes the real Uranium tilesets to GBA 4bpp and writes the
`engine/**/uranium_*.gen.h` tileset fragments (S8a); converts each map's layout
to `map.bin`/`border.bin` (S8b); compiles every `.pory` with poryscript into
`engine/data/maps/<Dir>/scripts.inc` + `CommonEvents.inc`, copies `map.json` +
layout bins into `engine/`, and writes the `*.gen.json` overlays + alias/flag
headers (S8c). Touches only gitignored generated paths under `engine/` — the
vendored tree's tracked files never change.

### 5. Build the ROM

```
make -C engine -j$(nproc) modern
```

Produces `engine/pokeemerald.gba`. If a previous build left stale generated
state (e.g. after switching between the playable slice and the map-walker debug
build below), `make -C engine clean` first.

### Debug variant: the Map Walker (all ~199 maps, warps-only, no scripts)

Instead of steps 2–4, flip `URANIUM_MAP_WALKER` to `TRUE` in
`engine/include/config/uranium_walker.h` and run:

```
.venv/bin/python -m rpg2gba.pipeline phase5 --maps full
make -C engine -j$(nproc) modern
```

This builds a cursor-based map inspector (no NPCs/scripts, warp-to-any-map menu)
instead of the playable slice — useful for spot-checking art/warps across the
whole corpus. Flip the flag back to `FALSE` and re-run steps 2–5 to get back to
the playable slice.

### Verifying the result

`pytest` exercises the converters this recipe drives (some tests skip without
`RPG2GBA_URANIUM_SRC`/`RPG2GBA_POKEEMERALD` set). Booting the ROM itself
requires an emulator (mGBA) — see `reference/engine_extension_surface.md` for
the engine-side divergences baked into this build (spawn override, intro skip,
player sprite, rock-smash test harness) if something looks wrong in-game.

## Reviewing map art (the Map Viewer)

A browser tool for eyeballing how a map converts to GBA art without booting the
ROM — the metatile layout, the RMXP→GBA layer collapse, and the palette
quantization. Needs step 1 of the ROM recipe above to have run at least once
(reads `output/uranium-build/maps/` + `tilesets.json`; it does not regenerate
them). Full reference: `reference/map_viewer.md`.

```
.venv/bin/python scripts/map_viewer_server.py --port 8765
```

Open `http://localhost:8765/` (it binds `0.0.0.0`, so a phone on the same
Tailscale/LAN works too). The landing page is a searchable, name+id index of
every map, nested under its RMXP parent; each row links to two views per map:

- **`/map/<id>`** — the zoomable map grid. Layer radio (RMXP composite / single
  RMXP layer / GBA composite / single GBA layer / post-quantization) plus
  overlay checkboxes (collision, diff, priority, palette-merge heatmap, events,
  warps). Click a cell for a per-layer breakdown in the sidebar.
- **`/palettes/<id>`** — the palette inspector: every GBA sub-palette's 15
  swatches and which metatiles draw from it, with suspect (≤2 colours) and bad
  (high merge-loss) tiles highlighted.

For offline/phone use without a running server, build self-contained HTML
instead (larger files, no cross-map nav):

```
.venv/bin/python scripts/build_map_viewer.py 32 49 48   # specific maps
.venv/bin/python scripts/build_map_viewer.py --all       # every map
```

Writes to `output/map_viewer/` (gitignored — don't commit it).

### Leaving feedback on a map

The map-grid view's sidebar has an **Issues** panel for flagging specific cells
while you review:

1. Click a cell on the canvas to select it.
2. Click **Flag selected cell** — a note box opens.
3. Type a note (optional) and click **Save**. The cell is now marked flagged
   (shows in the sidebar as "⚠ Flagged: <note>") and appears in the Issues list
   below, with a **✕** to remove it.

In **server mode**, Save and the ✕ delete write straight through to
`output/uranium-build/map_viewer_issues/MapNNN.json` (gitignored) — no export
step needed, and reloading the page reloads whatever's already saved for that
map. That file is a plain `{"x,y": {x, y, note}, ...}` object, so once you've
gone through a map and flagged what looks wrong, `cat` (or hand) that file to
Claude for the actual fixes. There's also an **Export JSON** button that
downloads a snapshot as an array — handy for the static/offline build below,
which has no server to persist to and keeps issues in-memory-only for that
reason.
